CREATING SERVICE-REGISTRY
- Created simple maven project with spring initializer, added dependancy of spring-boot-starter-webmvc,
spring-cloud-starter, spring-cloud-starter-netflix-eureka-server in pom.xml
- Also added @EnableEurekaServer, for telling & adding characteristics of Eureka server
- As we do not want service-registry itself to be registered somewhere else (avoid Cyclic dependancy) 
- In service-registry (server) application.yml configuration, 
eureka:
  instance:
    hostname: localhost  --“This service is running on localhost”
  client:
    register-with-eureka: false  --“Do NOT register this service in Eureka server”
    fetch-registry: false  --"Do NOT fetch other services from Eureka"
    service-url:
      defaultZone: http://localhost:8761/eureka/   --“Eureka server is running at this URL”
- Clients use this to:
    Register themselves
    Fetch other services
-All other services, add extra dependancy related to eureka client - spring-cloud-starter,
spring-cloud-starter-netflix-eureka-client.
- Also added @EnableDiscoveryClient, for telling & adding characteristics of Eureka client
-In other services (client) application.yml configuration, 
eureka:
  instance:
    prefer-ip-address: true
  client:
    service-url:
      defaultZone: http://localhost:8761/eureka/
- To enable a particular spring-cloud-starter dependancy, needs to be added with spring-cloud.version
and dependencyManagement,
<properties>
		<java.version>17</java.version>
		<spring-cloud.version>2025.1.0</spring-cloud.version>
</properties> and
<dependencyManagement>
    <dependencies>
      <dependency>
        <groupId>org.springframework.cloud</groupId>
        <artifactId>spring-cloud-dependencies</artifactId>
        <version>${spring-cloud.version}</version>
        <type>pom</type>
        <scope>import</scope>
      </dependency>
    </dependencies>
</dependencyManagement>

IMPLEMENTING CIRCUIT BREAKER PATTERN
- In USER-SERVICE, circuit breaker is added
- Add dependancy actuator, aop and resilience4j-spring-boot3 - to enable annotations and 
classes
- In target controller's method, use @CircuitBreaker & maintain fallback method
@CircuitBreaker(name = "userServiceCircuitBreaker", fallbackMethod = "fetchUserByIdFallback")
- Fallback method should have same return type and parameters as the original method along 
with an additional parameter of type Throwable to capture the exception details
FLOW:
  1. Failure Decision:
  minimumNumberOfCalls: 5
  failureRateThreshold: 50
  👉 Meaning:
  Wait for at least 5 calls
  If 50% fail, then → OPEN circuit
  📌 Easy memory:
  “5 calls → 50% fail → open”

  2. Sliding Window:
  slidingWindowSize: 10
  slidingWindowType: COUNT_BASED
  👉 Meaning:
  Check last 10 requests
  Based on count (not time)
  📌 Memory:
  “Check last 10 calls”

  3. Open State Timing:
  waitDurationInOpenState: 6s
  👉 Meaning:
  Stay OPEN for 6 seconds
  No calls allowed during this time
  📌 Memory:
  “Fail → sleep 6 sec”

  4. Half-Open Testing:
  permittedNumberOfCallsInHalfOpenState: 3
  👉 Meaning:
  After 6 sec → allow only 3 test calls
  📌 Memory:
  “After sleep → test 3 calls”

  5. Auto Transition:
  automaticTransitionFromOpenToHalfOpenEnabled: true
  👉 Meaning:
  Automatically move from OPEN → HALF-OPEN
  (no manual trigger needed)

  6. Monitoring / Metric:
  registerHealthIndicator: true
  eventConsumerBufferSize: 10
  👉 Meaning:
  Expose health info (for Actuator)
  Store last 10 events (for debugging)

RETRY MECHANISM USING resilience4j
Request comes
→ Retry tries 3 times
→ Failures counted in Circuit Breaker
→ If threshold crossed → Circuit OPEN
→ Fallback executed

Next request:
→ Circuit OPEN
→ No retry
→ Direct fallback

- Use @Retry in controller method, which you are targeting
    @Retry(name = "userServiceRetry", fallbackMethod = "fetchUserByIdFallback")
- Maintain different fallback method of circuitbreaker and retry mechanism
- Fallback is executed ONLY after:
    - Retry is exhausted OR
    - Circuit is OPEN
- Maintain configuration:
    resilience4j:
        retry:
        instances:
        userServiceRetry:
            maxAttempts: 3  #maximum 3 times (1 original + 2 retries)
            waitDuration: 2s  #Wait 2 seconds between retries
            retryExceptions:
            - java.lang.IllegalStateException  #Retry only for allowed Exceptions
            ignoreExceptions:
            - java.lang.NullPointerException  #Don't retry for this Exceptions


RATE LIMITER USING resilience4j
- Helps in security and performance of application
- Security: does not allow spam/bots to call service multiple times unnecessarily, leading
to server down
-Performace: Makes application always available to serve genuine requests from Clients, improved
scalabilty, reliability
- Need, same dependancy as CircuitBreaker - aop and resilience4j-spring-boot3
@RateLimiter(name = "userServiceRateLimiter", fallbackMethod = "rateLimitFetchUserByIdFallback")
- Max 5 calls
- Per 1 minute
- No waiting (fail fast)
- Configuration:
    resilience4j:
        rateLimiter:
            instances:
            userServiceRateLimiter:
                limitForPeriod: 5  # Max 5 calls per period
                limitRefreshPeriod: 1m  # in every 1 minute
                timeoutDuration: 0  # No waiting, fail immediately if limit exceeded - (fail fast)
-Balanced configuration:
    limitForPeriod: 10
    limitRefreshPeriod: 1s
    timeoutDuration: 2s
